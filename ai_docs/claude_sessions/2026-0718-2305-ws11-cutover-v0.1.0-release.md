## Release bug: our tag crashed ghostty's build (commit `907ac06`)

First `v0.1.0` push ‚Üí release run failed on **all three** platforms in
`make vt`:

    thread NNN panic: tagged releases must be in vX.Y.Z format matching build.zig

**Cause:** the vendored ghostty `build.zig` runs `git describe`, which
walks up out of `third_party/libghostty-vt` into the herdr-web repo. On a
tag-triggered checkout it saw `v0.1.0`, treated it as a *ghostty* release
tag, and panicked because it doesn't match ghostty's own version
(`1.3.2-dev` in `build.zig.zon`). Branch CI never hit it because
`actions/checkout` on branches fetches `--no-tags`; only tag checkouts see
the tag. Reproduced locally the moment the tag existed.

**Fix:** `scripts/build-libghostty-vt.sh` now passes
`-Dversion-string=1.3.2-dev` ‚Äî an explicit version makes ghostty's
`Config.zig` skip git detection entirely (the only other skip path is
`is_dep`; a `VERSION` file would NOT help ‚Äî it changes the expected
version but git detect still runs).

## Gotchas / techniques

- **Any vX.Y.Z tag in this repo breaks `make vt`** without the
  version-string pin ‚Äî the pin is now in the script; don't remove it.
- Permission classifier blocks tag deletion / force-push even chained;
  hand those to the user (`! cmd` bash-input works well).
- CI log fetch: token via `git credential fill`, then
  `Authorization: Bearer` on `actions/jobs/<id>/logs` (no gh CLI).
- Background poll loop on `actions/runs/<id>` (`until status=completed;
  sleep 45`) + run_in_background notifies on completion.
- gopls diagnostics show phantom "undefined: orchestration.NewHost" ‚Äî it
  lacks `-tags ghostty`; trust `make check`.
